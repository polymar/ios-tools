Multiline Comment

/**
 * Comment explaining method
 */
- (id) myMethod;

Inline Comment

/** Comment explaining method. */
- (id) myMethod; 


Content

Directly explain what the method does. Avoid being verbose and redundant.

Example, DO NOT DO THIS

/**
 * This class is the data model for a Car
 */
@interface MyClass : …. 

/** This method is used to get the value of x */
- (id) myMethod; 

But, DO THIS. The position of the comment already tells that you are referring to that method or class.

/**
 * Car data model
 */
@interface MyClass : ….  

/** Get the value of x */
- (id) myMethod;  

What to Comment
The following items must ALWAYS have a related comment (multiline or inline is up to the developer).

  • Constants defined in an interface file
  • Class definition in an interface file
  • Every method and property defined in an interface file
  • Protocol definition in an interface file
  • Every method defined inside a protocol definition
  • Class extension and Categories, in both interface file and implementation file (NOTE: documentation will not be generated for Class extension or Categories defined inside an implementation file, but comments are required to understand the purpose of the module)
  • Every method defined inside a Class extension or Category
  • Any other piece of code you feel deserves comments for better readability and maintainability
